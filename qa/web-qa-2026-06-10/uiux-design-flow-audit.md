## 2. Product Design Reading

FactTracer의 제품 경험은 “팩트체크 도구”보다 “정확한 보도와 사건 전말을 정리하는 편집실”에 가깝다. 사용자는 특정 주장 하나의 참/거짓만 확인하려는 것이 아니라, 한 사건이 어떻게 전개됐고, 어떤 사실이 확인됐고, 어떤 의견과 해석이 충돌하는지 한 번에 이해하려 한다.

현재 화면은 의도적으로 신문형 레이아웃과 리포트형 상세 구성을 사용한다. 이 방향은 정확한 보도 서비스의 매체성과 신뢰감을 만드는 데 맞다. 따라서 이 감사는 “신문형 구성을 바꿔야 한다”는 의미가 아니라, 그 의도된 구성 안에서 사건 흐름, 전말 요약, 이해관계자별 주장, 의견 충돌, 근거 확인 상태가 충분히 드러나는지를 보는 관점이다.

디자인 판단 기준:

- 최상위 정보: 사건의 전말, 현재까지 확인된 사실, 아직 갈리는 해석.
- 핵심 흐름: 사건 발견 → 전개 흐름 파악 → 의견 충돌 이해 → 근거/보도 비교 → 저장/알림/리포트/자료 제보.
- 금지해야 할 방향: 단일 주장 판정기, 일반 뉴스 포털, 일반 SaaS 대시보드, 원시 시스템 상태 노출.

## 4. Strengths

### 4.1 브랜드 톤이 제품 목적과 맞다

홈과 상세는 신문/리포트 톤을 사용한다. 큰 세리프 로고, 얇은 구분선, 넓은 여백, 제한된 색상은 “정밀 보도 매체” 느낌을 만든다. 신뢰 서비스에 맞는 방향이다.

### 4.2 상세 화면의 정보 골격은 좋다

이슈 상세는 `핵심 요약`, `확인된 사실`, `쟁점 지도`, `주장 검증`, `기사 비교`, `관점별 주장`, `타임라인`, `원문 자료`, `참여/제보`로 나뉜다. 새 방향 기준으로 보면 사건 전말, 주장 충돌, 보도 비교, 근거 확인을 한 화면 구조로 옮긴 점은 적절하다.

### 4.3 데스크톱 상세의 좌측 섹션 내비게이션은 긴 문서 탐색에 유용하다

상세 페이지가 긴 리포트형 구조라 좌측 섹션 내비게이션은 사용자가 위치를 잃지 않게 돕는다.

### 4.4 모바일은 가로 페이지 스크롤 없이 렌더링된다

모바일 홈과 이슈 상세는 전체 페이지 폭을 밀지 않는다. 표나 긴 콘텐츠는 내부 스크롤 또는 잘린 영역으로 처리되어 기본 반응형 구조는 안정적이다.

## 5. UX Risks

### 5.1 의도된 뉴스형 홈 안에서 “사건 전말” 신호가 더 강해질 수 있다

증거: `30-current-home-desktop-full.png`, `46-current-home-mobile-viewport.png`

홈의 첫 인상은 의도된 주요 기사/분야별 기사 편집면이다. 이 구성 자체는 정확한 보도 서비스 방향과 맞다. 다만 카드가 “이 사건의 전말을 정리했다”는 신호보다 일반 기사 카드처럼 먼저 읽힌다. `검증 진행 중` 배지보다 더 중요한 것은 `무슨 일이 있었고`, `누가 무엇을 주장하며`, `어디서 의견이 갈리는지`의 압축 신호다.

개선:

- 주요 사건 카드에 `사건 흐름`, `주요 충돌`, `확인된 사실`, `추가 확인` 요약을 작게 추가한다.
- `검증 진행 중`만 두지 말고 `전말 정리 중`, `의견 충돌 3개`, `공식 근거 5개 반영`처럼 보도 편집 상태를 보여준다.
- 현재 편집면 구조는 유지하되, 카드 안에 “정확한 보도/사건 분석”의 메타 정보를 더 선명하게 만든다.

### 5.2 헤더 검색은 기대와 다르게 작동하지 않는다

증거: `31-current-header-search-result.png`

데스크톱 헤더에 검색창이 있지만 Enter 제출 후 결과가 갱신되지 않았다. 검색은 정보 탐색 제품의 가장 강한 진입점이라 실패 영향이 크다.

개선:

- Enter 제출과 검색 아이콘 클릭을 모두 지원한다.
- 검색 후 `/?q=검색어` 이동, 입력값 유지, 결과 수 표시를 제공한다.
- 결과 없음 상태에는 `자료 제보/사건 보강 요청으로 보내기` CTA를 연결한다.

### 5.3 분야 필터 후 카테고리 레이블과 내용이 어긋난다

증거: `32-current-topic-economy.png`

`?topic=경제` 상태에서 `분야별 사건`의 `정치`, `사회` 칸 아래 경제 이슈가 들어간다. 사용자는 분류 신뢰도를 의심하게 된다.

개선:

- 필터 상태에서는 `분야별 사건` 섹션을 숨기거나 `경제 관련 사건` 단일 섹션으로 바꾼다.
- 고정 카테고리 레이블을 유지하려면 각 카테고리별 쿼리를 별도로 유지한다.

### 5.4 자료 제출/추가 확인 요청 실패가 사용자 흐름을 끊는다

증거: `40-current-verify-after-submit.png`

검증 요청 실패가 `Failed to fetch`로 노출된다. 현재 서비스 방향에서는 이 폼이 단순 팩트체크 요청이라기보다 보도를 보강할 자료/링크/본문을 보내는 진입점이므로, 실패하면 사용자는 사건 보도에 참여하는 흐름을 완료하지 못한다. 네트워크 문제인지, 서버 문제인지, 입력 문제인지, 로그인 필요인지도 알 수 없다.

개선:

- `요청을 보낼 수 없습니다. 연결 상태를 확인한 뒤 다시 시도해 주세요.`처럼 사용자 언어로 변환한다.
- 실패 후 입력값을 유지하는 것은 좋다. 여기에 `다시 시도`, `내용 복사`, `이슈 모니터로 돌아가기`를 제공한다.
- 성공 시에는 접수 번호, 처리 단계, 해당 사건에 연결될지 여부, 내 요청 확인 경로를 보여준다.

### 5.5 저장/알림/리포트 생성 CTA가 상태를 충분히 설명하지 않는다

증거: `33-current-issue-desktop-top.png`, `36-current-issue-after-save.png`, `38-current-issue-report-timeout.png`

상세 상단 CTA는 작고 간결해서 좋지만, 실패/성공/로그인 필요 상태가 명확하지 않다. 특히 저장은 사용자에게 반복 사용 가치가 큰 액션이라 즉시 피드백이 필요하다.

개선:

- 비로그인 저장 클릭: `저장하려면 로그인이 필요합니다` + `로그인` CTA.
- 리포트 생성 실패: 원인별 메시지와 직접 리포트 보기 대안 제공.
- 성공: 버튼 라벨을 `저장됨`, `리포트 열기`로 전환한다.

## 6. Design And Layout Risks

### 6.1 홈의 뉴스형 시각 언어는 유지하되 사건 분석 메타 정보를 보강하면 좋다

증거: `30-current-home-desktop-full.png`

이미지 카드가 강하고 사건 분석 상태는 작다. 의도된 신문형 편집을 유지하더라도, FactTracer의 본질이 “정확한 보도로 사건 전말과 충돌 의견을 정리하는 것”이라면 카드마다 사건 흐름과 충돌 요약이 제목 다음의 고정 정보로 보여야 한다.

개선:

- 카드 하단에 작은 사건표를 넣는다: `흐름 5단계`, `충돌 3개`, `근거 5개`, `보류 2개`.
- 이미지 크기는 유지하되 상태/근거 정보를 이미지 아래 고정 위치에 둔다.
- “최신 검증” 사이드바는 `최신 보도 업데이트` 또는 `최근 정리된 사건`에 가깝게 바꾸고, 판정 변화뿐 아니라 새 국면/새 반론/공식 출처 갱신을 강조한다.

### 6.2 상세 첫 화면은 리포트형 의도가 잘 보이며, 사건 전말 미리보기를 더할 여지가 있다

증거: `33-current-issue-desktop-top.png`

상단에서 제목/요약/CTA 후 핵심 팩트 테이블이 먼저 보이는 구성은 리포트형 상세 의도와 맞다. 다만 “정확한 보도”가 핵심이라면 독자는 먼저 사건의 전말과 주요 국면을 기대한다. 현재 구조를 유지하면서도 상단에 전개 흐름과 의견 충돌 미리보기를 얹으면 탐색 비용이 줄어든다.

개선:

- 상단 요약 아래에 `사건 흐름 3-5단계`와 `의견 충돌 2-3개` 미리보기를 추가한다.
- `쟁점 지도` 섹션은 유지하되, 상세 첫 화면 안에 “핵심 충돌” 요약 링크나 미리보기를 둔다.
- `확인된 사실`과 `미해결 질문`을 같은 행에 배치해 판단 유보를 명확히 한다.

### 6.3 기사 비교표는 기능적으로 맞지만 읽기 부담이 크다

증거: `35-current-issue-articles-section.png`

표가 넓고 컬럼이 많아 리서처에게는 유용하지만, 일반 사용자에게는 “어디를 먼저 봐야 하는지”가 약하다.

개선:

- 표 위에 요약 칩을 추가한다: `업데이트 필요 2건`, `공식 근거 없음 8건`, `서로 다른 수치 1건`.
- 기본 정렬을 위험도/업데이트 필요순으로 둔다.
- 모바일에서는 표 대신 카드형 비교 요약을 우선 보여주고, 전체 표는 펼침으로 둔다.

### 6.4 리포트 페이지는 목적형 액션이 좋지만 내부 이벤트명이 남아 있다

증거: `45-current-report-page.png`

`PDF 저장`, `인용 복사`, `비교표 복사`, `출처 목록 복사`는 학생/리서처에게 적합하다. 하지만 `new_article` 같은 내부 이벤트명이 보이면 리포트 신뢰감이 떨어진다.

개선:

- `new_article` → `새 기사 반영`
- `verification_update` → `판정 변경`
- 리포트에서는 시스템 이벤트명 전체를 사용자 언어로 변환한다.

## 7. Accessibility Risks

### 7.1 빈 링크가 주요 사건 카드에 존재한다

증거: `30-current-home-desktop-full.png`의 DOM 확인 결과, 주요 사건 카드에 텍스트 없는 링크가 반복된다.

위험:
- 스크린리더 사용자는 목적을 알 수 없는 링크를 만난다.
- 키보드 사용자는 같은 카드에서 링크 포커스를 두 번 밟을 수 있다.

개선:

- 이미지 링크에 `aria-label="{이슈 제목} 상세 보기"`를 넣거나 카드 전체를 하나의 링크로 합친다.
- 중복 링크가 필요하면 하나는 `tabIndex={-1}` 또는 `aria-hidden` 처리한다.

### 7.2 섹션 내비게이션이 `tab` 의미를 사용하지만 실제는 문서 섹션 전환이다

증거: `33-current-issue-desktop-top.png`, DOM 확인 결과 데스크톱/모바일용 `role=tab` 세트가 모두 존재.

위험:
- 화면에는 하나만 보여도 보조기술에서는 중복 탭 목록처럼 해석될 수 있다.
- 섹션 내비게이션이면 `nav + button/link`가 더 자연스럽다.

개선:

- 실제 패널 전환 UI라면 보이는 tablist만 DOM/접근성 트리에 남긴다.
- 단순 섹션 이동이라면 `role=tab` 대신 링크 목록으로 구현한다.
- 숨긴 breakpoint 내비게이션에는 `hidden`/`aria-hidden` 상태가 접근성 트리에 반영되게 한다.

### 7.3 모바일 카테고리 nav가 잘려 보이고 검색이 사실상 사라진다

증거: `46-current-home-mobile-viewport.png`

상단 카테고리 nav는 가로 스크롤이고 `과학/기술`이 잘린 채 보인다. 검색창은 모바일에서 보이지 않아 사용자는 하단 탭과 카테고리 nav에만 의존한다.

개선:

- 모바일 헤더에 검색 아이콘 버튼을 두고 전체 검색을 열 수 있게 한다.
- 가로 nav에는 끝부분 fade 또는 `더보기`를 둬 스크롤 가능성을 명확히 한다.
- 현재 선택된 분야는 더 강하게 표시한다.

### 7.4 오류 메시지와 상태 변화가 보조기술에 충분히 전달되는지 불확실하다

증거: `40-current-verify-after-submit.png`

오류는 시각적으로 빨간 텍스트지만, 현재 캡처만으로는 `aria-live`나 focus 이동이 적절한지 확인할 수 없다.

개선:

- 실패/성공 메시지 영역에 `role="status"` 또는 `aria-live="polite"`를 적용한다.
- 제출 실패 시 오류 요약으로 focus를 이동하거나, 다음 Tab 순서에서 즉시 만날 수 있게 한다.

## 8. Screen-Level Recommendations

### Home

우선순위:

1. 검색을 작동시킨다.
2. 주요 사건 카드에 사건 흐름/의견 충돌 지표를 추가한다.
3. 분야 필터 상태에서 카테고리 레이블 불일치를 없앤다.
4. 모바일 검색 진입점을 추가한다.

### Issue Detail

우선순위:

1. 상단에 `사건 흐름`과 `핵심 충돌 3개` 미리보기를 추가한다.
2. 저장/알림/리포트 CTA에 성공/실패/로그인 필요 상태를 명확히 둔다.
3. 섹션 내비게이션의 `tab` 의미를 실제 동작에 맞게 정리한다.
4. 기사 비교표 위에 요약 지표를 제공한다.

### Verify

우선순위:

1. `Failed to fetch`를 사용자 언어로 변환한다.
2. 성공 시 접수 번호와 사건 연결/보도 보강 진행 단계가 남게 한다.
3. 모드 카드의 텍스트 줄바꿈을 정리한다. 현재 일부 좁은 카드에서 `YouTube`가 세로로 깨져 보인다.
4. 모바일에서는 모드 선택을 2열 또는 segmented control로 압축한다.

### Report

우선순위:

1. 내부 이벤트명을 제거한다.
2. 복사 버튼에 `복사됨` 피드백을 추가한다.
3. 중복 원문 링크를 그룹화한다.
4. 리포트 상단에 `발표용 요약`, `토론 쟁점`, `인용 출처` 바로가기 구조를 둔다.

### Admin

우선순위:

1. 비로그인 상태에서는 운영 폼보다 권한 안내를 먼저 보여준다.
2. 빈 상태는 `무엇을 확인해야 하는지`보다 `다음으로 할 수 있는 작업`을 더 직접적으로 제공한다.
3. 운영 화면은 공개 화면과 더 분리된 밀도와 상태 색 체계를 가져도 된다.

## 9. Recommended User Flow Reframe

현재 흐름:

`홈에서 매체형으로 탐색 → 상세에서 리포트처럼 읽기 → CTA 클릭 → 실패/불명확`

권장 흐름:

`홈에서 매체형 편집을 훑으면서 사건 전말과 의견 충돌 신호도 스캔 → 상세에서 사건 흐름, 핵심 충돌, 확인 사실을 확인 → 근거/기사 비교로 확장 → 저장/알림/리포트/자료 제보 중 하나를 완료 → 완료 상태와 다음 경로 확인`

화면 구성 관점의 핵심 변경:

- 홈은 현재 뉴스형 편집 의도를 유지하되, 카드 안에 사건 흐름과 의견 충돌을 압축하는 메타 정보를 더해야 한다.
- 상세 첫 화면은 현재 리포트형 구조를 유지하되, “사건 전말/현재 국면/의견 충돌/보류 지점”의 미리보기를 더 빠르게 보여줘야 한다.
- 리포트는 긴 원문 덤프가 아니라 사건 전말, 근거, 의견 충돌을 재사용할 수 있는 보도 리포트 단위로 정리되어야 한다.
- 모든 쓰기 액션은 성공/실패/진행 중/로그인 필요 상태가 화면에 남아야 한다.

## 10. Evidence Limits

- 스크린샷과 DOM 관찰 기반 감사이므로 완전한 WCAG 준수 여부는 판단하지 않았다.
- 실제 키보드 순서, 스크린리더 발표문, 색 대비 수치, 확대 200% 상태는 별도 테스트가 필요하다.
- 현재 `localhost:3002`는 CORS 문제로 클라이언트 mutation이 실패하므로 성공 후 상태는 충분히 확인하지 못했다.
