# FactTracer UI/UX QA Persona Synthesis

검증일: 2026-06-09  
기준 문서: `PRD.md`  
검증 대상: 현재 로컬 서비스 UI, 사용자 흐름, 주요 기능 제공 여부  
검증 방식: PRD 분석, 로컬 UI 브라우저 점검, acceptance audit 실행, 사용자 페르소나별 하위 세션 피드백 수집 후 종합

## 1. 결론

현재 서비스는 "정치/사회 이슈를 구조화하고 팩트 중심으로 이해시킨다"는 방향성은 전달한다. 홈, 이슈 상세, 저장, 검증 요청, 관리자 화면의 큰 뼈대도 갖춰져 있다.

다만 PRD의 핵심 가치인 **쟁점 지도**, **관점별 주장 정리**, **라벨링 없는 안전한 표현**, **출처 기반 검증 흐름**, **사용자 목적별 리포트/알림/저장 흐름**은 아직 UI에서 충분히 드러나지 않는다. 사용자는 "검증된 팩트 리스트를 보는 서비스"로는 이해할 수 있지만, "복잡한 이슈의 주장 구조를 비교하고 목적에 맞게 활용하는 서비스"로 쓰기에는 흐름과 기능이 부족하다.

가장 큰 문제는 다음 4가지다.

1. **쟁점 지도와 관점별 주장 흐름이 약하다.** 현재 쟁점 지도는 충돌하는 주장 맵이라기보다 개별 팩트 카드 목록에 가깝다.
2. **사용자 언어보다 시스템 언어가 많이 노출된다.** `2026-06-09T...`, `신뢰도 0.52`, `news_search`, 개발 오버레이성 문구가 신뢰감을 떨어뜨린다.
3. **페르소나별 핵심 목적 달성 CTA가 불분명하다.** 일반 사용자는 알림, 학생은 발표용 리포트, 연구자는 비교표/출처 내보내기, 관리자는 검토 큐와 승인 절차가 더 전면에 있어야 한다.
4. **검증 요청 이후와 관리자 검토 이후의 상태 흐름이 끊긴다.** 요청 접수, 재검증, 보류, 반려, 승인, 게시 같은 상태가 사용자가 추적 가능한 워크플로우로 보이지 않는다.

## 3. 페르소나별 종합 피드백

### 3.1 일반 뉴스 사용자

목적: 긴 이슈를 빠르게 이해하고, 관심 이슈를 저장하거나 업데이트를 받는 것.

좋은 점:
- 전체 톤은 차분하고 신뢰 기반 서비스처럼 보인다.
- 홈에서 이슈를 클릭해 상세로 들어가는 흐름은 자연스럽다.
- 상세 페이지의 핵심 팩트, 기사, 자료, 타임라인 구조는 초심자에게도 큰 방향을 준다.

문제:
- 첫 화면에서 "기사 검증 URL 제출"과 "이슈 읽기"가 동시에 강조되어 초보자는 무엇을 먼저 해야 할지 헷갈릴 수 있다.
- `신뢰도 0.52`, 원시 timestamp는 일반 사용자에게 의미가 불명확하다.
- "이슈 알림 받기" 흐름이 약해 PRD의 관심 이슈 추적 가치가 충분히 보이지 않는다.
- 검증 요청 후 "접수되었습니다"만 보이고 이후 어디서 확인하는지 알기 어렵다.
- 모바일 하단 탭은 명확하지만 데스크톱 헤더에는 핵심 경로가 부족하다.

개선 방향:
- 홈의 1순위는 "지금 이해해야 할 이슈"로 두고, 기사 URL 제출은 보조 CTA로 낮춘다.
- `신뢰도 0.52`는 `출처 일치도 보통`, `검증 근거 3개 확인`처럼 사용자 언어로 바꾼다.
- 검증 요청 완료 후 `내 검증 요청 보기`, `진행 상태 알림 받기`를 제공한다.
- 데스크톱에도 `홈`, `저장`, `알림`, `검증 요청` 같은 명확한 전역 내비게이션을 제공한다.

### 3.2 정치/사회 이슈 관심 사용자

목적: 특정 이슈에서 어느 쪽이 무엇을 주장하고, 무엇이 사실이며, 무엇이 미확인인지 구분하는 것.

좋은 점:
- 자극적인 정치 UI가 아니라 차분한 검증 서비스 톤을 유지한다.
- 사실 카드와 출처 중심 구성은 감정적 소비를 줄이는 데 도움이 된다.

문제:
- PRD의 핵심인 "쟁점 지도"가 현재는 실제 지도라기보다 팩트 카드 목록처럼 보인다.
- `사실`, `대체로 사실` 배지가 강하게 보이고, `검증 어려움`, `의혹`, `반론`, `공통분모` 같은 민감 이슈용 상태가 부족하다.
- 관점별 주장이 독립 섹션이나 탭으로 보이지 않는다.
- 사용자가 이슈 구조를 이해하기 전에 주장 제출 폼이 먼저 눈에 들어온다.
- 현재 주장 제출 유형은 PRD의 `사실 주장`, `해석/평가`, `요구 사항`, `운동 전략`, `반박`, `추가 근거 제보`와 맞지 않는다.

개선 방향:
- 이슈 상세의 핵심 구조를 `핵심 팩트 -> 쟁점 지도 -> 관점별 주장 -> 반론/공통분모 -> 근거 자료`로 재정렬한다.
- 쟁점 지도는 `주장 A`, `주장 B`, `충돌 이유`, `확인된 사실`, `미확인 주장`, `공통분모`가 한눈에 보이도록 재설계한다.
- 낙인성 표현은 `근거 부족`, `검증 불가`, `결론 유보`, `공식 확인 전` 같은 안전한 상태로 처리한다.
- 주장 제출 폼은 사용자가 맥락을 읽은 뒤 진입하도록 상세 하단이나 별도 CTA로 낮춘다.

### 3.3 학생/토론/발표 사용자

목적: 이슈를 발표, 토론, 과제 자료로 빠르게 정리하고 근거를 인용하는 것.

좋은 점:
- 핵심 팩트, 기사 비교, 공식 자료, 타임라인, 수치 변화 같은 섹션은 발표 자료의 기본 골격으로 적합하다.
- 상세 페이지의 섹션 앵커는 긴 글을 탐색하는 데 도움이 된다.

문제:
- 현재 화면은 "발표에 바로 쓸 수 있는 리포트"라기보다 긴 검증 페이지에 가깝다.
- `리포트 저장`, `리포트 보기`가 무엇을 저장/표시하는지 모호하다.
- 출처와 타임라인이 단순 목록으로 흩어져 있어 발표 논리로 재구성하기 어렵다.
- 상세 하단에 읽기를 마친 뒤 할 수 있는 `발표용 리포트 보기`, `인용 복사`, `PDF 저장` 같은 목적형 액션이 부족하다.
- 섹션 네비게이션에 현재 위치 표시가 약해 긴 페이지에서 길을 잃기 쉽다.

개선 방향:
- 리포트 CTA를 `발표용 리포트 보기`, `출처 포함 PDF 저장`, `인용 복사`처럼 목적 기반으로 바꾼다.
- 학생용 리포트는 `요약`, `찬반/쟁점`, `근거`, `반박`, `타임라인`, `참고자료` 구조로 제공한다.
- 상세 페이지 하단에 "이 이슈로 발표 자료 만들기" 흐름을 추가한다.
- `요약만 보기`와 `전체 근거 보기` 같은 읽기 모드를 제공한다.

### 3.4 기자/연구자/콘텐츠 제작자

목적: 기사나 콘텐츠 제작을 위해 출처, 수치, 업데이트 이력, 기사 간 차이를 빠르게 검증하는 것.

좋은 점:
- 팩트, 기사, 출처, 타임라인, 수치 변화가 모두 별도 섹션으로 존재해 기본 정보 구조는 맞다.
- 차분하고 도구적인 톤은 연구/검증 업무에 어울린다.

문제:
- 기사 비교가 연구자가 원하는 비교표 형태가 아니라 세로 목록에 가깝다.
- 어떤 기사 문장이 어떤 공식 자료나 수치에 근거했는지 추적하기 어렵다.
- `공식 자료`, `언론 보도`, `통계`, `법령`, `2차 자료` 같은 출처 등급이 시각적으로 구분되지 않는다.
- 수치 변화 섹션은 값의 변화, 기준 시점, 근거 링크를 연결해서 보여주지 못한다.
- `리포트 저장`보다 `출처 포함 리포트`, `비교표 저장`, `인용용 내보내기`가 이 페르소나에게 더 적합하다.

개선 방향:
- 기사 비교를 표로 바꾼다: `기사`, `발행 시각`, `포함 주장`, `사용 수치`, `공식 출처 인용 여부`, `업데이트 필요`, `판정`.
- 출처 카드에는 `공식`, `언론`, `통계`, `법령`, `자료집` 같은 배지를 붙인다.
- 수치 변화는 `이전 값 -> 현재 값 -> 근거 -> 반영 시점` 흐름으로 보여준다.
- 연구자 액션으로 `원문 열기`, `인용 복사`, `출처 목록 저장`, `비교표 내보내기`를 제공한다.

### 3.5 관리자/검토자

목적: 신고, 사용자 주장, 출처 업데이트, 검증 결과를 안전하게 검토하고 게시하는 것.

좋은 점:
- 관리자 화면의 큰 범위는 존재한다. 이슈, 신고, 출처, 자동 처리 기록의 축은 PRD와 방향이 맞다.
- 검토 업무를 별도 관리자 영역으로 분리한 점은 적절하다.

문제:
- 현재 관리자 화면은 검토 워크벤치보다 일반 대시보드처럼 보인다.
- 업무 큐, 위험도, 실패 상태, 공식 출처 업데이트 필요 여부가 한눈에 드러나지 않는다.
- 상세 화면에서 `완료된 검토`, `남은 검토`, `보류 사유`, `게시 전 체크`가 보이지 않는다.
- 신고 처리 흐름은 단순 처리/기각에 가까우며, 표현 정제나 사유 기록이 부족하다.
- 게시/승인처럼 위험한 액션에 확인 모달, 체크리스트, 승인 메모가 부족하다.
- 권한 없음, 데이터 없음, 일시 실패, 로그인 필요 상태가 시각적으로 충분히 구분되지 않는다.

개선 방향:
- 관리자 홈은 카드형 요약보다 `검토 큐` 중심으로 재설계한다.
- 이슈 상세에 `수집`, `AI 초안`, `출처 확인`, `주장 검토`, `표현 정제`, `게시 승인` 단계 표시를 둔다.
- 게시 전에는 `공식 출처 확인`, `낙인 표현 제거`, `반론 확인`, `보류 사유 없음` 체크리스트와 승인 메모를 요구한다.
- 신고 처리에는 `문제 표현`, `대체 표현`, `판단 사유`, `처리 결과`를 남기는 폼을 제공한다.

## 4. 공통 UX 문제

### 4.1 시스템 언어가 신뢰를 떨어뜨림

원시 timestamp, 소수점 신뢰도, 내부 출처 키, 개발 오버레이성 문구는 사용자에게 "아직 제품이 완성되지 않았다"는 인상을 준다. 신뢰 서비스에서는 특히 치명적이다.

권장:
- 날짜: `2026년 6월 9일 15:22 업데이트`
- 신뢰도: `출처 일치도 보통`, `근거 3개 중 2개 일치`
- 출처 유형: `공식 발표`, `언론 보도`, `통계 자료`
- 상태: `검증 중`, `공식 확인 전`, `결론 유보`, `업데이트 필요`

### 4.2 PRD 핵심 기능의 시각적 우선순위가 낮음

PRD에서 중요한 것은 단순 팩트 판정이 아니라 "주장 충돌과 관점 구조를 정리하는 것"이다. 그러나 현재 상세 페이지는 팩트 카드, 기사 목록, 자료 목록이 중심이고 쟁점 지도/관점별 주장은 약하다.

권장:
- 이슈 상세 상단에 `이 이슈의 핵심 쟁점 3개`를 먼저 보여준다.
- 쟁점별로 `주장`, `반론`, `확인된 사실`, `아직 모르는 것`, `공통분모`를 묶는다.
- 관점별 주장은 독립 섹션 또는 탭으로 제공한다.

### 4.3 목적별 CTA가 섞여 있음

일반 사용자, 학생, 연구자, 관리자가 원하는 다음 행동은 다르다. 현재 CTA는 `리포트 저장`, `검증 요청`, `주장 제출`처럼 넓고 모호하다.

권장:
- 일반 사용자: `이슈 저장`, `업데이트 알림 받기`
- 학생: `발표용 리포트 보기`, `인용 복사`, `PDF 저장`
- 연구자: `비교표 저장`, `출처 포함 내보내기`, `원문 열기`
- 정치/사회 관심 사용자: `관점별 주장 보기`, `반론 보기`, `공통분모 보기`
- 관리자: `검토 시작`, `보류`, `재검증 요청`, `승인 게시`

### 4.4 요청 후 상태 추적이 약함

검증 요청이나 주장 제출은 제출 자체보다 "이후 어떻게 처리되는지"가 더 중요하다. 현재는 완료 메시지 이후 상태 추적 흐름이 약하다.

권장:
- 제출 완료 화면에 `접수됨 -> 수집 중 -> 검증 중 -> 검토 완료 -> 반영됨/반려됨` 상태를 표시한다.
- 사용자의 `내 검증 요청`, `내 주장 제보` 화면으로 연결한다.
- 알림 설정과 연결한다.

## 5. 권장 통합 사용자 흐름

### 5.1 일반 사용자 공개 흐름

1. 홈에서 오늘의 주요 이슈와 최근 업데이트를 본다.
2. 관심 이슈를 선택한다.
3. 상세 상단에서 핵심 요약과 업데이트 시점을 확인한다.
4. 쟁점 지도에서 주장 충돌 구조를 파악한다.
5. 관점별 주장과 확인된 사실, 미확인 내용을 비교한다.
6. 필요하면 기사 비교, 출처, 타임라인, 수치 변화를 펼쳐 본다.
7. 이슈를 저장하거나 업데이트 알림을 설정한다.
8. 부족한 주장/근거가 있으면 하단에서 구조화된 제보를 제출한다.

### 5.2 학생/연구자 활용 흐름

1. 이슈 상세에서 `발표/연구 모드`를 선택한다.
2. 요약, 쟁점, 근거, 반론, 타임라인을 목적별 레이아웃으로 본다.
3. 기사 비교표와 출처 목록을 확인한다.
4. 인용 문구와 원문 링크를 복사한다.
5. 출처 포함 리포트나 비교표를 저장한다.

### 5.3 관리자 검토 흐름

1. 관리자 홈에서 위험도와 최신 업데이트 기준으로 검토 큐를 본다.
2. 이슈 상세에서 AI 초안, 사용자 주장, 신고, 출처 업데이트를 함께 확인한다.
3. 주장별로 사실/해석/요구/반박/추가 근거를 분류한다.
4. 공식 출처와 반론 확인 여부를 체크한다.
5. 낙인 표현이나 불확실 표현을 정제한다.
6. 보류/반려/재검증/승인을 선택하고 판단 사유를 기록한다.
7. 게시 전 체크리스트를 완료한 뒤 반영한다.

## 6. 우선순위 백로그

### P0: PRD 핵심 가치 회복

1. 쟁점 지도 재설계: 주장 A/B, 충돌 이유, 확인된 사실, 미확인 내용, 공통분모를 한 화면에서 보여준다.
2. 관점별 주장 섹션 추가: PRD의 주장 유형에 맞춰 사실 주장, 해석/평가, 요구 사항, 운동 전략, 반박, 추가 근거 제보를 분리한다.
3. 시스템 언어 제거: 원시 timestamp, 소수점 신뢰도, 내부 출처 키, 개발 오버레이성 문구를 사용자 언어로 바꾼다.
4. 제출 후 상태 추적 추가: 검증 요청과 주장 제출 이후의 진행 상태를 볼 수 있게 한다.

### P1: 사용자 목적별 완성도 개선

1. 홈 정보 위계 조정: 이슈 이해를 1순위, 기사 URL 검증 제출을 2순위로 둔다.
2. 데스크톱 전역 내비게이션 추가: 홈, 저장, 알림, 검증 요청, 계정 진입을 명확히 한다.
3. 리포트 UX 재정의: 발표용, 출처 포함, 기사 비교표 등 목적별 리포트 CTA로 나눈다.
4. 기사 비교표 추가: 기사 간 주장, 수치, 출처, 업데이트 필요 여부를 표로 비교한다.
5. 출처 배지 체계 추가: 공식, 언론, 통계, 법령, 2차 자료를 구분한다.
6. 관리자 워크벤치 개선: 검토 큐, 단계 표시, 승인 체크리스트, 판단 메모를 추가한다.

### P2: 사용성 및 신뢰감 개선

1. 긴 상세 페이지에 현재 섹션 표시를 추가한다.
2. `요약만 보기`와 `전체 근거 보기` 읽기 모드를 제공한다.
3. 비어 있음, 권한 없음, 로그인 필요, 일시 실패 상태를 서로 다르게 디자인한다.
4. 주장 제출 폼을 상세 맥락 하단으로 낮추고 PRD의 유형 체계와 맞춘다.
5. 저장/알림/리포트 완료 후 다음 행동을 명확히 제공한다.

## 9. 최종 판단

현재 UI는 PRD의 방향성을 부분적으로 구현했지만, 사용자 입장에서는 아직 **"팩트 카드와 자료 목록을 보는 서비스"**에 더 가깝다. PRD가 의도한 **"복잡한 정치/사회 이슈를 주장 구조, 관점, 근거, 변화 이력으로 안전하게 이해시키는 서비스"**가 되려면 쟁점 지도, 관점별 주장, 목적별 리포트, 알림/상태 추적, 관리자 검토 워크플로우를 우선적으로 강화해야 한다.

가장 먼저 손봐야 할 화면은 이슈 상세다. 이 화면에서 쟁점 구조와 관점별 주장이 제대로 보이면 일반 사용자, 정치/사회 관심 사용자, 학생, 연구자 모두의 목적 달성률이 함께 올라간다. 그 다음은 제출 후 상태 추적과 관리자 검토 워크벤치다.
