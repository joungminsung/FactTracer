# PRD: FactTracer / 팩트트레이서

## 1. 문서 개요

### 제품명

FactTracer / 팩트트레이서

### 제품 유형

AI 기반 실시간 이슈 검증 및 주장 구조화 플랫폼

### 플랫폼

* Web
* Mobile App
* PWA
* Admin Dashboard

### 제품 한 줄 정의

팩트트레이서는 이슈 되는 기사들을 자동 수집하고, 기사 속 주장·수치·근거를 검증한 뒤, 사람들의 다양한 해석과 요구를 쟁점별로 구조화해 보여주는 AI 기반 이슈 검증 플랫폼이다.

---

# 2. 제품 배경

## 2.1 문제 상황

뉴스, 유튜브, SNS, 커뮤니티에서는 특정 사건이 발생하면 수많은 기사와 주장이 빠르게 쏟아진다.

예를 들어 선거 관련 논란, 정책 발표, 지원금, 통계, 재난, 집회, 사회적 갈등 이슈가 발생하면 다음과 같은 문제가 생긴다.

* 같은 사건에 대해 기사마다 다른 수치를 사용한다.
* 초기 보도와 후속 보도의 정보가 섞인다.
* 공식자료가 나오기 전 추측성 주장이 확산된다.
* 사실, 해석, 의혹, 전략 주장이 뒤섞인다.
* 특정 진영이나 집단을 낙인찍는 표현이 확산된다.
* 사람들은 상대 입장이 왜 그런 주장을 하는지 이해하지 못한다.
* 댓글창은 논쟁보다 비난과 분탕으로 흐르기 쉽다.

기존 팩트체크 서비스는 특정 주장 하나를 검증하는 데 집중한다. 하지만 실제 사용자가 궁금해하는 것은 단일 기사 하나가 아니라 다음에 가깝다.

> 지금 이 사건에서 무엇이 사실이고, 어떤 주장이 충돌하고 있으며, 사람들은 왜 다르게 해석하는가?

# 3. 제품 비전

## 3.1 제품 비전

팩트트레이서는 단순한 “가짜뉴스 판별기”가 아니다.

팩트트레이서는 다음 세 가지를 동시에 수행한다.

1. **팩트 검증**

   * 기사, 공식자료, 통계, 보도자료를 기반으로 사실 여부를 검증한다.

2. **쟁점 구조화**

   * 같은 사건 안에서 충돌하는 주장, 수치, 해석, 요구를 정리한다.

3. **관점 이해**

   * 사람들의 입장을 진영 라벨이 아니라 주장 유형과 쟁점 중심으로 정리해 서로 이해할 수 있게 한다.

## 3.2 핵심 철학

```text
팩트는 검증하고,
의견은 분리하고,
쟁점은 구조화하고,
낙인은 정제한다.
```

## 3.3 제품 포지셔닝

기존 서비스와 비교하면 다음과 같다.

| 구분     | 기존 뉴스 서비스 | 일반 팩트체크 | 팩트트레이서        |
| ------ | --------- | ------- | ------------- |
| 중심 단위  | 기사        | 개별 주장   | 사건/이슈         |
| 업데이트   | 언론사별      | 수동 검증   | 자동 수집·자동 갱신   |
| 검증 단위  | 없음        | 주장      | 주장·수치·근거      |
| 의견 정리  | 댓글        | 거의 없음   | 쟁점 지도         |
| 사용자 참여 | 댓글/공감     | 제보 수준   | 구조화된 주장 제출    |
| 목표     | 정보 전달     | 사실 확인   | 사실 검증 + 입장 이해 |

# 4. 핵심 제품 구조

## 4.1 정보 계층

팩트트레이서는 다음 계층으로 정보를 관리한다.

```text
Topic
└─ Issue
   └─ Claim Cluster
      └─ Claim
         └─ Evidence
```

### Topic

큰 분야.

예시:

* 선거
* 정치
* 경제
* 교육
* 복지
* 재난
* 의료
* 사회
* 지역 이슈

### Issue

실제로 사용자가 보는 핵심 단위.

예시:

* 선관위 투표용지 부족 사태
* 청년 지원금 정책 논란
* 전기요금 인상 보도
* 대학 입시 제도 개편 논란
* 특정 지역 재난 대응 논란

### Claim Cluster

같은 쟁점에 속하는 주장 묶음.

예시:

* 부족 투표소 수
* 원인
* 책임 소재
* 법적 쟁점
* 재발 방지 대책
* 운동 전략

### Claim

기사, 사용자, 공식자료에서 나온 개별 주장.

예시:

* “전국 50곳에서 투표용지가 부족했다.”
* “단순 행정 실수다.”
* “재선거를 요구해야 한다.”
* “부정선거 의혹까지 제기해야 한다.”

### Evidence

주장을 검증하거나 뒷받침하는 근거.

예시:

* 공식 보도자료
* 선관위 발표
* 정부 정책자료
* 통계자료
* 언론 보도
* 법령
* 판례
* 현장 영상
* 사용자 제보 자료

# 5. 제품 목표

## 5.1 사용자 목표

사용자는 팩트트레이서를 통해 다음을 할 수 있어야 한다.

* 지금 이슈 되는 사건을 빠르게 파악한다.
* 어떤 수치가 최신인지 확인한다.
* 기사마다 어떤 주장을 담고 있는지 비교한다.
* 공식자료와 기사 내용의 차이를 확인한다.
* 서로 다른 입장의 핵심 논리를 이해한다.
* 자신의 주장을 근거와 함께 제출한다.
* 단순 댓글 싸움이 아니라 쟁점 중심으로 토론한다.

## 5.2 서비스 목표

* 이슈 되는 기사 자동 감지
* 관련 기사 자동 수집
* 새 기사 등장 시 준실시간 업데이트
* 주장·수치·날짜·기관 자동 추출
* 주장별 근거 매칭
* 기사별 오류/업데이트 필요 지점 표시
* 쟁점별 입장 정리
* 유저 참여형 주장 구조화
* 분탕과 낙인 표현 최소화
* 웹과 앱 모두에서 사용 가능

# 6. 주요 사용자

## 6.1 일반 사용자

### 니즈

* 지금 논란이 뭔지 빠르게 알고 싶다.
* 어떤 기사가 맞고 어떤 기사가 오래된 정보인지 알고 싶다.
* 복잡한 이슈를 한눈에 정리해서 보고 싶다.

### 사용 방식

* 홈에서 실시간 이슈 확인
* 이슈 페이지 열람
* 기사별 검증 결과 확인
* 리포트 공유

---

## 6.2 정치/사회 이슈 관심 사용자

### 니즈

* 특정 사건의 사실관계를 정확히 알고 싶다.
* 상대 입장이 왜 그런 주장을 하는지 보고 싶다.
* 내 주장을 근거와 함께 정리하고 싶다.

### 사용 방식

* 쟁점 지도 확인
* 주장 스펙트럼 확인
* 구조화된 주장 제출
* 다른 입장과의 충돌 지점 확인

---

## 6.3 학생/토론/발표 사용자

### 니즈

* 이슈를 객관적으로 정리하고 싶다.
* 찬반이 아니라 쟁점과 근거 중심으로 보고 싶다.
* 발표나 토론 자료로 활용하고 싶다.

### 사용 방식

* 핵심 팩트 요약 확인
* 주장별 근거 확인
* 쟁점 지도 활용
* 리포트 저장

---

## 6.4 기자/리서처/콘텐츠 제작자

### 니즈

* 여러 기사에서 수치와 근거가 어떻게 달라지는지 보고 싶다.
* 원문 자료와 기사 표현의 차이를 확인하고 싶다.
* 새로운 공식 입장이 나왔는지 추적하고 싶다.

### 사용 방식

* 사건 타임라인 확인
* 기사별 주장 비교
* 원문 근거 확인
* 업데이트 로그 확인

---

## 6.5 관리자/검토자

### 니즈

* 자동 분석 품질을 관리해야 한다.

* 출처 신뢰도와 사용자 신고를 관리해야 한다.

### 사용 방식

* 관리자 대시보드
* 에이전트 로그 확인
* 신고 처리
* 출처 도메인 관리
* 재검증 실행

# 7. 핵심 기능 요약

## 7.1 자동 이슈 감지

시스템은 뉴스 수집원을 주기적으로 확인하여 급상승 이슈를 감지한다.

### 감지 기준

* 짧은 시간 내 기사 수 증가
* 여러 언론사 동시 보도
* 수치 포함 여부
* 공식기관 언급 여부
* 선거, 정책, 재난, 교육, 복지 등 고영향 분야 여부
* SNS/검색량 증가
* 기존 이슈와 유사한 기사 증가

### 예시

```text
키워드: 투표용지 부족
1시간 기사 수: 43개
언론사 수: 12개
공식기관 언급: 선관위
수치 포함: 14곳, 50곳, 4만 장
이슈 점수: 91
결과: 자동 이슈 생성
```

---

## 7.2 기사 자동 수집

이슈로 판단된 키워드에 대해 관련 기사를 자동 수집한다.

### 수집 대상

* 뉴스 검색 API
* 언론사 RSS
* 공식기관 보도자료
* 정부/공공기관 게시판
* 기존 팩트체크 결과
* 유튜브 영상 제목/설명/자막
* SNS 공개 게시글 후보

### 수집 주기

* 고위험/급상승 이슈: 1~5분 간격
* 일반 이슈: 10~30분 간격
* 낮은 우선순위 이슈: 1~3시간 간격

### 중복 제거 기준

* URL
* 제목 유사도
* 본문 유사도
* 언론사
* 발행 시간
* 핵심 문장 임베딩 유사도

---

## 7.3 사건/이슈 페이지 자동 생성

시스템은 이슈 점수가 기준 이상이면 자동으로 이슈 페이지를 생성한다.

### 이슈 페이지 기본 구성

* 이슈명
* 현재 상태
* 마지막 업데이트 시간
* 관련 기사 수
* 검증된 주장 수
* 추가 확인 필요 주장 수
* 핵심 팩트 요약
* 쟁점 지도
* 주장별 검증
* 기사별 비교
* 관점별 주장
* 타임라인
* 원문 자료

## 7.4 주장 자동 추출

각 기사에서 검증 가능한 주장을 추출한다.

### 추출 대상

* 수치 주장
* 날짜 주장
* 기관 발표 주장
* 정책 내용 주장
* 법적 주장
* 원인 주장
* 책임 주장
* 요구 사항
* 운동 전략 주장
* 의혹 주장

### 제거 또는 별도 분리 대상

* 감정 표현
* 단순 비난
* 모욕
* 선동성 문구
* 근거 없는 낙인
* 단순 전망
* 확인 불가능한 주장

## 7.5 주장 유형 분류

모든 주장은 다음 유형 중 하나 이상으로 분류한다.

| 유형    | 설명                 | 예시                    |
| ----- | ------------------ | --------------------- |
| 사실 주장 | 검증 가능한 사실          | “전국 50곳에서 부족했다.”      |
| 수치 주장 | 숫자 포함 주장           | “투표용지 4만 장이 남았다.”     |
| 원인 해석 | 원인에 대한 해석          | “투표율 예측 실패 때문이다.”     |
| 책임 주장 | 책임 소재 주장           | “선관위 책임이 크다.”         |
| 법적 주장 | 법적 판단 관련 주장        | “재선거 사유다.”            |
| 요구 사항 | 행동 요구              | “재선거를 해야 한다.”         |
| 운동 전략 | 어떤 메시지를 외칠지에 대한 주장 | “부정선거보다 재선거만 외쳐야 한다.” |
| 의혹 주장 | 고의성, 조작, 침투 등      | “특정 세력이 방해했다.”        |
| 낙인 표현 | 상대를 비난하는 표현        | “쁘락지다.”               |

---

## 7.6 주장 클러스터링

같은 쟁점에 속하는 주장을 묶는다.

### 예시: 투표용지 부족 사태

```text
클러스터 1: 부족 투표소 수
- 부족 투표소는 14곳이다.
- 전국 50곳에서 부족했다.
- 일부 투표소에서 부족했다.

클러스터 2: 원인
- 투표율 예측 실패다.
- 배분 실패다.
- 선관위 관리 부실이다.

클러스터 3: 요구 사항
- 재선거가 필요하다.
- 진상조사가 먼저다.
- 책임자 문책이 필요하다.

클러스터 4: 운동 전략
- 재선거 요구로 메시지를 모아야 한다.
- 부정선거 의혹까지 외쳐야 한다.
- 참정권 침해 프레임으로 가야 한다.
```

## 7.7 공식자료 기반 검증

주장별로 근거 자료를 검색하고 매칭한다.

### 우선순위

1. 공식기관 발표
2. 정부/부처 보도자료
3. 중앙선관위/공공기관 자료
4. 법령/판례/국회 자료
5. 국가통계/KOSIS
6. 공공데이터
7. 기존 팩트체크 결과
8. 복수 언론 보도
9. 기관 공식 SNS
10. 개인 블로그/커뮤니티

### 검증 항목

* 날짜
* 기관명
* 수치
* 정책명
* 대상
* 장소
* 법적 상태
* 공식 발표 여부
* 기사 작성 시점
* 후속 보도 반영 여부

## 7.8 판정 체계

팩트트레이서는 단순히 “진짜/가짜”로 판정하지 않는다.

### 판정 라벨

| 라벨       | 의미                      |
| -------- | ----------------------- |
| 사실       | 공식자료와 핵심 내용이 일치함        |
| 대체로 사실   | 핵심은 맞지만 세부 차이가 있음       |
| 일부 사실    | 일부 주장은 맞고 일부는 다름        |
| 초기 기준    | 작성 당시에는 맞았으나 후속 정보가 있음  |
| 업데이트 필요  | 최신 자료가 반영되지 않았을 가능성이 있음 |
| 맥락 누락    | 중요한 조건, 시점, 대상이 빠짐      |
| 과장       | 원자료보다 강하게 표현됨           |
| 오해 소지    | 사실이지만 독자가 다르게 이해할 수 있음  |
| 근거 부족    | 확인 가능한 근거가 부족함          |
| 단정 불가    | 가능성은 있으나 결론 내릴 수 없음     |
| 법적 판단 필요 | 자동 판정이 어려운 법적 쟁점        |
| 사실 아님    | 신뢰도 높은 근거와 명확히 충돌함      |
| 검증 불가    | 현재 자료로 판단 불가능           |

---

## 7.9 기사별 검증

기사 전체를 “가짜뉴스”라고 판정하지 않는다.

기사별로 다음을 보여준다.

* 해당 기사가 포함한 주장
* 각 주장별 검증 상태
* 최신 수치 반영 여부
* 공식자료 인용 여부
* 기사 작성 시점 기준 적절성
* 후속 업데이트 필요 여부
* 다른 기사와의 차이점

### 예시

```text
기사 A

포함된 주장:
- 부족 투표소 14곳
- 선관위 사과 예정
- 일부 투표 지연

분석:
- 14곳 수치는 초기 파악 기준으로 보임
- 후속 조사 수치가 반영되지 않음
- 원인 단정은 없음

상태:
업데이트 필요
```

## 7.10 주장 지도 / 쟁점 지도

이슈 페이지에는 쟁점 지도를 제공한다.

### 목적

* 입장을 보수/진보로 나누지 않는다.
* 사람을 분류하지 않고 주장을 분류한다.
* 충돌 지점을 명확히 보여준다.
* 서로 다른 입장의 핵심 논리를 이해하게 한다.

### 예시

```text
쟁점 1. 원인
A. 투표율 예측 실패
B. 현장 배분 실패
C. 선관위 관리 부실
D. 고의성 의혹

쟁점 2. 요구 사항
A. 재선거 요구
B. 진상조사 요구
C. 책임자 문책
D. 제도 개선

쟁점 3. 운동 전략
A. 재선거 구호 집중
B. 부정선거 의혹 제기
C. 참정권 침해 프레임
D. 법적 절차 중심
```

## 7.11 관점별 주장 정리

사용자와 기사에서 나온 입장을 AI가 요약해 관점별로 묶는다.

### 금지

* “보수 의견”
* “진보 의견”
* “이 사람은 보수”
* “이 사람은 진보”
* “특정 진영이 거짓말한다”

### 권장

* “관리 책임 강조 관점”
* “선거 불신 확대 경계 관점”
* “참정권 침해 강조 관점”
* “부정선거 의혹 제기 관점”
* “재선거 요구 집중 관점”
* “법적 절차 중심 관점”

### 예시

```text
관점 A: 관리 책임 강조

핵심 주장:
- 선관위의 현장 관리 실패가 핵심 문제다.
- 책임자 문책과 제도 개선이 필요하다.
- 단순 실수로 넘기면 안 된다.

검증 가능한 근거:
- 투표용지 부족 발생 보도
- 선관위 사과
- 후속 조사 필요성

검증 어려운 부분:
- 고의성 여부
- 조직적 방해 여부
```

## 7.12 낙인 표현 처리

사용자나 기사에서 낙인 표현이 등장할 수 있다.

예시:

* “쁘락지다”
* “조작 세력이다”
* “선동꾼이다”
* “빨갱이다”
* “부정선거충이다”

서비스는 이런 표현을 그대로 확산하지 않는다.

### 처리 방식

#### 원문

```text
재선거만 외치라는 사람들은 쁘락지다.
```

#### 정제 표현

```text
일부 참여자는 ‘재선거’ 구호로 메시지를 제한하는 움직임이 문제의 본질을 약화시킨다고 보고 있다.
```

#### 검증 상태

```text
상대 세력 침투 또는 조직적 방해 주장: 현재 확인 가능한 근거 부족
```

## 8.1 자유 댓글 금지 또는 제한

팩트트레이서는 일반 댓글창을 핵심 기능으로 두지 않는다.

### 기본 입력 필드

```text

[추가 근거 제보]
```

---

## 8.3 사용자 성향 선택 금지

공개적으로 보수/진보/중도 성향을 선택하게 하지 않는다.

### 이유

* 위장 선택 가능
* 분탕 가능
* 진영 낙인 발생
* 주장보다 라벨 중심 공격 가능

### 대안

사용자는 자신의 정치 성향이 아니라 관심 관점을 선택할 수 있다.

예시:

```text
관심 관점:
[관리 책임]
[참정권 침해]
[법적 쟁점]
[제도 개선]
[언론 보도 문제]
[전체 관점 보기]
```

이 정보는 개인화에만 사용하며 공개하지 않는다.


# 11. AI 에이전트 설계

## 11.1 Agent 목록

### 1. News Watcher Agent

새 뉴스와 보도자료를 감지한다.

### 2. Issue Detector Agent

기사 증가량, 키워드, 공식기관 언급, 수치 포함 여부를 기반으로 이슈성을 판단한다.

### 3. Issue Matcher Agent

새 기사가 기존 이슈에 속하는지 판단한다.

### 4. Article Parser Agent

기사 제목, 언론사, 날짜, 본문을 추출한다.

### 5. Claim Extractor Agent

검증 가능한 주장을 추출한다.

### 6. Entity Extractor Agent

날짜, 기관, 장소, 수치, 인물, 정책명 등을 추출한다.

### 7. Claim Clusterer Agent

같은 쟁점에 속하는 주장을 묶는다.

### 8. Evidence Retriever Agent

공식자료, 통계, 기존 팩트체크, 언론 보도 등을 검색한다.

### 9. Evidence Ranker Agent

출처 신뢰도와 관련도를 평가한다.

### 10. Claim Verifier Agent

주장과 근거를 비교해 판정한다.

### 11. Perspective Mapper Agent

주장들을 관점과 쟁점별로 정리한다.

### 12. Toxicity & Label Filter Agent

낙인, 비방, 혐오, 근거 없는 의혹을 정제한다.

### 13. Report Builder Agent

사용자에게 보여줄 리포트를 생성한다.

### 14. Audit Logger Agent

모든 분석 과정을 기록한다.

# 12. 기능 요구사항

## 12.1 자동 이슈 기능

### FR-ISSUE-001

시스템은 뉴스 데이터를 주기적으로 수집해야 한다.

### FR-ISSUE-002

시스템은 기사 증가량, 출처 다양성, 수치 포함 여부, 공식기관 언급 여부를 기준으로 이슈 점수를 계산해야 한다.

### FR-ISSUE-003

이슈 점수가 기준 이상이면 자동으로 이슈 후보를 생성해야 한다.

### FR-ISSUE-004

관리자는 이슈 후보를 승인, 병합, 삭제할 수 있어야 한다.

### FR-ISSUE-005

고신뢰 이슈는 자동 공개될 수 있으며, 민감 이슈는 검토 후 공개되어야 한다.

---

## 12.2 기사 수집 기능

### FR-ARTICLE-001

시스템은 이슈와 관련된 기사를 자동 수집해야 한다.

### FR-ARTICLE-002

시스템은 중복 기사를 제거해야 한다.

### FR-ARTICLE-003

시스템은 기사 본문 추출 실패 시 제목/요약 기반 분석 또는 수동 입력을 지원해야 한다.

### FR-ARTICLE-004

사용자는 기사 링크를 직접 제보할 수 있어야 한다.

---

## 12.3 주장 추출 기능

### FR-CLAIM-001

시스템은 기사에서 검증 가능한 주장을 자동 추출해야 한다.

### FR-CLAIM-002

시스템은 주장을 사실, 수치, 해석, 요구, 전략, 의혹, 낙인 표현으로 분류해야 한다.

### FR-CLAIM-003

시스템은 같은 쟁점에 속하는 주장을 클러스터링해야 한다.

### FR-CLAIM-004

시스템은 새로운 주장이 기존 클러스터에 속하지 않으면 새 클러스터 후보로 등록해야 한다.

---

## 12.4 검증 기능

### FR-VERIFY-001

시스템은 주장별로 공식자료와 근거 자료를 검색해야 한다.

### FR-VERIFY-002

시스템은 주장과 근거의 날짜, 수치, 기관명, 장소, 대상, 상태를 비교해야 한다.

### FR-VERIFY-003

시스템은 주장별 판정 라벨을 생성해야 한다.

### FR-VERIFY-004

시스템은 검증 불가나 근거 부족을 허용해야 한다.

### FR-VERIFY-005

시스템은 공식자료가 새로 등장하면 관련 주장 판정을 재계산해야 한다.

---

## 12.5 쟁점 지도 기능

### FR-MAP-001

시스템은 이슈별 주요 쟁점을 자동 생성해야 한다.

### FR-MAP-002

시스템은 쟁점 안의 주장들을 관점별로 요약해야 한다.

### FR-MAP-003

시스템은 관점 간 충돌 지점과 공통분모를 표시해야 한다.

### FR-MAP-004

시스템은 보수/진보 라벨 대신 주장 중심 관점명을 사용해야 한다.

---

## 12.6 사용자 참여 기능

### FR-USER-CLAIM-001

사용자는 구조화된 주장 제출 폼을 통해 의견을 제출할 수 있어야 한다.

### FR-USER-CLAIM-002

사용자는 근거 링크를 함께 제출할 수 있어야 한다.

### FR-USER-CLAIM-003

시스템은 사용자 주장을 자동 분류하고 기존 주장과 중복 여부를 확인해야 한다.

### FR-USER-CLAIM-004

낙인, 비방, 근거 없는 의혹은 정제 또는 비공개 처리되어야 한다.

### FR-USER-CLAIM-005

사용자의 정치 성향은 공개 라벨로 사용하지 않는다.

---

## 12.7 알림 기능

### FR-NOTI-001

사용자는 관심 이슈를 구독할 수 있어야 한다.

### FR-NOTI-002

이슈에 새 공식자료, 새 수치, 판정 변경이 발생하면 알림을 받을 수 있어야 한다.

### FR-NOTI-003

앱은 푸시 알림을 지원해야 한다.

---

## 12.8 직접 검증 기능

### FR-MANUAL-001

사용자는 뉴스 URL을 직접 입력해 검증을 요청할 수 있어야 한다.

### FR-MANUAL-002

사용자는 텍스트를 직접 입력해 검증을 요청할 수 있어야 한다.

### FR-MANUAL-003

사용자는 이미지, PDF, 유튜브 링크를 입력할 수 있어야 한다.

### FR-MANUAL-004

## 12.9 관리자 기능

### FR-ADMIN-002

# 13. 비기능 요구사항

## 13.1 성능

* 급상승 이슈 감지 주기: 1~5분
* 일반 이슈 감지 주기: 10~30분
* 기사 수집 후 주장 추출 목표 시간: 1분 이내
* 새 주장 검증 목표 시간: 3~10분
* 이슈 페이지 업데이트 반영 목표 시간: 5분 이내
* 사용자가 직접 입력한 기사 검증 목표 시간: 1~5분

---

## 13.2 안정성

* 작업 큐 기반 비동기 처리
* 실패 작업 자동 재시도
* 일부 분석 실패 시 부분 리포트 제공
* 크롤링 실패 시 제목/요약 기반 분석 대체
* 공식자료 미발견 시 “근거 부족” 출력

---

## 13.3 투명성

* 모든 검증 결과는 근거 링크를 표시해야 한다.
* AI가 생성한 요약과 실제 근거를 분리해야 한다.
* 검증 불가 사유를 명확히 표시해야 한다.
* 판정 변경 이력을 기록해야 한다.
* 에이전트 로그를 관리자에게 제공해야 한다.

---

## 13.4 안전성

* 특정 개인, 집단, 정당, 진영에 대한 단정적 비난 방지
* 낙인 표현 정제
* 보수/진보 공개 라벨링 금지
* 근거 없는 고의성·침투·조작 주장은 “근거 부족” 처리
* 법적 판단은 “법적 판단 필요”로 분리
* 의료, 금융, 재난, 선거 등 고위험 이슈는 주의 문구 표시

---

## 13.5 보안

* 업로드 파일 확장자 및 MIME 타입 검사
* 사용자 데이터 계정별 분리
* 관리자 권한 role 기반 제한
* 사용자 제보 자료 비공개 기본값
* 신고 데이터 보존
* 악성 링크 필터링

# 19. 리스크 및 대응

## 19.1 사건 단위 업데이트가 느려 보일 위험

### 문제

사건 자체는 자주 바뀌지 않아 서비스가 정체되어 보일 수 있다.

### 대응

* 사건 안의 주장 단위 업데이트 표시
* 새 수치 등장 표시
* 공식자료 등장 표시
* 기사 추가 로그 표시
* 타임라인 업데이트 제공

---

## 19.2 기사 수가 적은 사건

### 문제

모든 사건이 기사 수십 개 이상 나오지는 않는다.

### 대응

* A급 이슈: 자동 이슈 페이지 생성
* B급 이슈: 작은 이슈 카드 생성
* C급 기사: 직접 검증 결과로 저장
* 기사 수보다 주장 변화와 공식자료 여부를 기준으로 평가

---

## 19.3 정치적 분탕

### 문제

유저 참여 기능이 진영 싸움으로 변질될 수 있다.

### 대응

* 자유 댓글 제한
* 구조화된 주장 제출
* 보수/진보 라벨 금지
* 낙인 표현 정제
* 근거 링크 요구
* 신고 및 관리자 검토

---

## 19.4 AI 환각

### 문제

AI가 없는 근거를 생성할 수 있다.

### 대응

* 근거 URL 없는 판정 금지
* 출처 기반 리포트 생성
* 에이전트 로그 저장
* 검증 불가 허용
* 공식자료 우선순위 적용

---

## 19.5 법적 리스크

### 문제

기사나 개인, 단체를 가짜뉴스로 단정하면 명예훼손 문제가 생길 수 있다.

### 대응

* 기사 전체 판정 금지
* 주장 단위 판정
* “현재 확인 가능한 근거 기준” 표현 사용
* 법적 쟁점은 자동 단정 금지
* 정정 요청/신고 기능 제공

---

## 19.6 공식자료 지연

### 문제

사건 직후에는 공식자료가 나오지 않을 수 있다.

### 대응

* “공식 확인 대기” 상태 표시
* 초기 보도/후속 보도 구분
* 판정 변경 이력 기록
* 공식자료 등장 시 재검증

# 20. 최종 제품 정의

팩트트레이서는 이슈 되는 기사들을 자동 수집하고, 기사 속 주장·수치·근거를 AI가 분석해 공식자료와 대조하는 실시간 이슈 검증 플랫폼이다.

서비스의 핵심 단위는 다음과 같다.

```text
사용자 관점: 사건별로 본다.
시스템 관점: 주장별로 추적한다.
데이터 관점: 기사별로 수집한다.
검증 관점: 근거별로 판단한다.
```

팩트트레이서는 단순히 가짜뉴스를 판별하지 않는다.

대신 다음을 수행한다.

* 사건을 자동 감지한다.
* 관련 기사를 자동 수집한다.
* 기사 속 주장을 추출한다.
* 같은 주장을 묶는다.
* 수치와 근거를 검증한다.
* 공식자료와 비교한다.
* 기사별 업데이트 필요 지점을 보여준다.
* 사람들의 입장을 쟁점별로 정리한다.
* 낙인과 분탕을 줄이고, 서로 다른 관점을 이해하게 한다.

최종적으로 팩트트레이서는 다음 문장으로 정의된다.

> 팩트트레이서는 실시간으로 이슈 되는 기사를 수집하고, 주장과 근거를 검증하며, 사람들의 다양한 입장을 쟁점별로 구조화해 보여주는 AI 기반 이슈 검증 및 관점 이해 플랫폼이다.
