# FactTracer Production Design Specification

이 문서는 `PRD.md`를 기준으로 FactTracer를 프로덕션 서비스로 만들기 위한 제품 설계, 화면 설계, 상태 설계, 운영 설계 기준을 정의한다. 현재 구현은 이 문서의 기준을 향해 맞춰야 하며, 빈 화면이나 내부 구현 설명을 제품 경험으로 대체해서는 안 된다.

## 1. 설계 목적

FactTracer는 뉴스 목록 서비스도, 단일 주장 판별기도 아니다. 사용자는 사건 단위로 쏟아지는 기사와 주장을 보고 다음 세 가지를 빠르게 판단해야 한다.

- 무엇이 확인된 사실인가.
- 어떤 주장과 수치가 충돌하는가.
- 서로 다른 관점이 어떤 근거를 사용하며 어디서 갈리는가.

프로덕션 설계의 목표는 PRD의 핵심 철학인 "팩트는 검증하고, 의견은 분리하고, 쟁점은 구조화하고, 낙인은 정제한다"를 화면 구조, 데이터 상태, 운영 플로우에 일관되게 반영하는 것이다.

## 2. 제품 정체성

### 2.1 제품 한 문장

FactTracer는 이슈가 된 사건을 기사, 주장, 쟁점, 근거 단위로 분해해 사용자가 사실 관계와 관점 차이를 한 화면에서 추적하게 하는 검증 플랫폼이다.

### 2.2 인터페이스 논지

이 인터페이스는 실시간 관제실이 아니라 정밀한 사건 기록대처럼 느껴져야 한다. 제품의 본질이 빠른 판정이 아니라 근거 있는 분해와 비교이기 때문에, 화면은 높은 정보 밀도, 명확한 계층, 절제된 상태 표현을 강조해야 한다.

### 2.3 의도된 느낌

- 정확함
- 차분함
- 긴장감 있음
- 검토 가능함
- 신뢰 가능함

### 2.4 피해야 할 느낌

- 랜딩 페이지처럼 설명하는 화면
- 카드 묶음으로만 보이는 일반 대시보드
- 빈 데이터 때문에 실패한 페이지처럼 보이는 화면
- 내부 기술 상태를 사용자에게 노출하는 개발용 화면
- 진영, 사람, 감정을 먼저 드러내는 커뮤니티 화면

## 3. 주요 사용자와 핵심 작업

| 사용자 | 핵심 작업 | 성공 기준 |
| --- | --- | --- |
| 일반 사용자 | 지금 이슈가 무엇인지 빠르게 파악 | 이슈 목록에서 상태, 변경, 핵심 사실을 즉시 이해 |
| 정치/사회 이슈 관심 사용자 | 서로 다른 주장의 근거와 충돌 지점 확인 | 쟁점 지도에서 관점별 주장과 검증 가능 지점 확인 |
| 학생/토론/발표 사용자 | 발표 가능한 구조로 이슈 정리 | 핵심 팩트, 원문, 타임라인, 리포트 저장 가능 |
| 기자/리서처 | 기사별 수치와 근거 차이 추적 | 기사 비교에서 오래된 정보와 최신 자료 차이 확인 |
| 관리자/검토자 | 자동 분석 결과 검수와 출고 | 검토 목록, 주장 검수, 출처 관리, 신고 처리, 재검증 완료 |

## 4. 핵심 정보 모델

PRD의 기본 계층을 프로덕션 화면과 API 계약의 공통 언어로 사용한다.

```text
Topic
└─ Issue
   └─ Claim Cluster
      └─ Claim
         └─ Evidence
```

### 4.1 Topic

큰 분야 필터다. 홈과 앱에서는 빠른 탐색 필터로 사용하며, 운영 콘솔에서는 위험도와 검토 우선순위의 보조 기준으로 사용한다.

### 4.2 Issue

사용자가 실제로 보는 기본 단위다. 기사나 댓글이 아니라 사건을 중심으로 한다. 모든 화면은 이슈가 현재 어떤 상태인지 먼저 보여야 한다.

### 4.3 Claim Cluster

같은 쟁점에 속하는 주장 묶음이다. 공개 화면에서는 "쟁점" 또는 "주장 묶음"으로 표현하고, 문서/코드/API 내부에서는 `ClaimCluster` 계약을 유지할 수 있다.

### 4.4 Claim

검증 가능한 개별 주장이다. 감정 표현, 낙인, 단순 전망은 Claim으로 직접 확산하지 않고 별도 정제 상태로 분리한다.

### 4.5 Evidence

주장을 확인하거나 반박하는 근거다. 출처 유형, 발행 시점, 신뢰도, 최신성, 공식성 여부가 함께 관리되어야 한다.

## 5. 제품 표면 구조

### 5.1 공개 웹

목적은 빠른 파악과 깊은 탐색이다.

필수 화면:

- 홈: 이슈 모니터, 토픽 필터, 감지된 사건, 선택 이슈 요약, 기사 검증 접수
- 이슈 상세: 핵심 팩트, 쟁점 지도, 기사 비교, 관점, 타임라인, 원문 자료, 주장 제출
- 로그인/회원가입: 계정 기반 저장, 검증 요청, 주장 제출 이력을 위한 인증
- 내 계정: 저장 이슈, 제출 주장, 검증 요청 이력

### 5.2 모바일 앱/PWA

목적은 짧은 시간 안에 확인하고 다시 돌아오게 하는 것이다.

핵심 탭:

- 홈: 관심 이슈와 급상승 이슈
- 이슈: 저장/팔로우한 이슈
- 검증하기: 공유된 기사 링크 또는 직접 입력
- 알림: 공식 자료, 수치 변경, 검토 완료 알림
- 내 정보: 저장, 제출, 알림 설정

모바일은 웹의 축소판이 아니라 알림과 재방문에 최적화되어야 한다. 상세 분석은 핵심 팩트와 쟁점 요약을 먼저 보여주고, 기사 비교와 원문 자료는 접힌 구조로 제공한다.

### 5.3 관리자 페이지

목적은 자동 분석 결과를 검수하고 공개 품질을 관리하는 것이다.

필수 화면:

- 운영 콘솔: 검토 목록, 민감 이슈, 신고 표현, 출처 관리, 자동 처리 상태
- 검토 상세: 주장 검수, 기사 비교, 신고 표현, 재검증, 출고 승인
- 신고 처리: 낙인 표현, 근거 없는 단정, 출처 신고 처리
- 출처 관리: 출처 신뢰도, 상태, 최근 검토 이력
- 자동 처리 기록: 수집, 추출, 근거 매칭, 낙인 필터링 상태

관리자 상세 화면은 데이터가 없거나 접근할 수 없어도 운영 콘솔의 골격을 유지해야 한다. 단독 에러 페이지처럼 보이면 안 된다.

## 6. 화면별 프로덕션 설계

### 6.1 홈

홈은 제품 소개가 아니라 작업 화면이어야 한다.

필수 영역:

- 상단 상태: 현재 수집 상태, 토픽 필터, 감지 이슈 수, 업데이트 수
- 기사 링크 입력: 기사 검증 접수
- 감지된 사건 목록: 제목, 분야, 기사 수, 쟁점 수, 변경 주장 수, 업데이트 시점
- 선택 이슈 본문: 핵심 요약, 현재까지 확인된 것, 충돌 주장 묶음, 근거 기준, 관점 이해
- 보조 레일: 업데이트가 있을 때만 표시, 선택 이슈가 있을 때만 주장 제출 표시

금지:

- 제품을 설명하는 큰 히어로 문구
- "AI 기반", "API 미연결", "서버 응답" 같은 내부 상태 문구
- 데이터가 없을 때 우측 보조 설명문만 남기는 화면

빈 상태:

- 이슈가 없으면 "표시할 이슈 없음"과 짧은 설명만 보여준다.
- 기사 검증 입력은 유지한다.
- 통계 영역은 0값으로 유지한다.
- 우측 업데이트/주장 제출 레일은 숨긴다.

### 6.2 이슈 상세

이슈 상세는 사건 기록의 중심 화면이다.

필수 상단 정보:

- 이슈명
- 분야, 위험도, 상태
- 요약
- 기사 수, 쟁점 수, 검증 완료 수, 추가 확인 필요 수, 업데이트 시점
- 저장, 리포트 저장, 기사 검증 접수

필수 본문:

- 핵심 팩트 요약
- 쟁점 지도
- 기사별 검증 비교
- 원문 자료
- 타임라인
- 주장 제출
- 관련 이슈

빈 상태:

- 특정 이슈를 찾을 수 없어도 헤더, 돌아가기, 검색/탐색 행동은 유지한다.
- "해당 이슈가 없거나 아직 공개되지 않았습니다"만 단독으로 놓지 않는다.
- 다음 행동: 홈으로 이동, 검증 요청, 최근 이슈 보기 중 최소 1개를 제공한다.

### 6.3 구조화된 주장 제출

사용자는 댓글이 아니라 검증 가능한 주장을 제출한다.

필드:

- 주장
- 이유
- 근거 링크
- 관련 쟁점
- 주장 유형
- 반박 가능 지점

처리 흐름:

```text
입력
↓
낙인 표현 필터링
↓
주장 유형 분류
↓
기존 쟁점과 유사도 비교
↓
중복이면 기존 관점 보강
↓
새 주장이면 검토 목록으로 이동
↓
관리자 승인 후 공개 반영
```

공개 화면에는 "댓글", "진영", "성향" 중심 UI를 두지 않는다.

### 6.4 로그인/회원가입/내 계정

인증 화면은 기능 설명 페이지가 아니라 사용 이력 진입점이다.

필수 정보:

- 저장 이슈
- 제출 주장
- 기사 검증 요청
- 알림/관심 이슈 설정
- 권한에 따른 운영 메뉴 접근

금지:

- 엔드포인트, 토큰, 서버, API 같은 내부 구현 문구 노출
- 개발 환경 설명

### 6.5 운영 콘솔

운영 콘솔은 일반 대시보드가 아니라 검수 작업대다.

필수 영역:

- 운영 메뉴
- 검토 목록
- 선택 검토 이슈
- 주장 검수
- 쟁점 판정
- 출처 기준
- 자동 처리 상태
- 재검증 실행
- 출고 전 확인

검토 목록 우선순위:

1. 고영향 분야
2. 공식 자료 갱신
3. 수치 충돌
4. 낙인 표현 신고 증가
5. 오래된 기사 확산
6. 관리자 재검증 요청

### 6.6 관리자 검토 상세

현재 구현에서 가장 프로덕트 느낌이 약한 화면이다. 상세 데이터를 불러오지 못할 때 화면 전체가 비어 보이는 것이 문제다.

프로덕션 기준:

- 운영 콘솔 헤더와 좌측 운영 메뉴를 유지한다.
- 검토 목록으로 돌아가는 경로를 유지한다.
- 상태를 구분한다: 없음, 비공개, 권한 없음, 삭제됨, 일시 실패.
- 다음 행동을 제공한다: 검토 목록으로 이동, 목록 갱신, 홈에서 이슈 검색.
- 정상 데이터가 있을 때와 같은 정보 골격을 일부 유지한다.

상태별 문구:

| 상태 | 사용자 문구 | 표시 행동 |
| --- | --- | --- |
| 없음 | 검토 대상을 찾을 수 없습니다 | 검토 목록으로 이동 |
| 비공개 | 아직 공개되지 않은 검토 항목입니다 | 권한 확인 또는 목록 이동 |
| 권한 없음 | 이 화면을 볼 권한이 없습니다 | 로그인 또는 계정 전환 |
| 삭제됨 | 이미 처리된 검토 항목입니다 | 처리 이력 보기 |
| 일시 실패 | 내용을 불러오지 못했습니다 | 다시 시도 |

### 6.7 신고 처리

신고 처리 화면은 낙인 표현과 근거 없는 단정을 줄이는 운영 도구다.

필수 정보:

- 신고 사유
- 대상 이슈
- 신고된 표현
- 우선순위
- 현재 처리 상태
- 처리/기각 액션

빈 상태는 "접수된 신고가 생기면 이 목록에 표시됩니다"처럼 짧게 유지한다.

### 6.8 출처 관리

출처 관리는 주장 검증 품질을 좌우한다.

필수 정보:

- 출처명
- 출처 주소 또는 식별자
- 유형: 공식, 언론, 통계, 법령, SNS, 커뮤니티
- 신뢰도
- 상태: 신뢰, 감시, 차단
- 최근 검토 시점
- 메모

사용자 화면에는 "도메인"보다 "출처"라고 표현한다.

### 6.9 자동 처리 기록

자동 처리 기록은 시스템이 무슨 판단을 했는지 검토자가 이해하는 화면이다.

필수 정보:

- 처리 종류: 수집, 중복 제거, 주장 추출, 근거 매칭, 낙인 필터링
- 대상
- 결과
- 실행 시각
- 재실행 액션
- 실패 사유

사용자 화면에는 "에이전트" 대신 "자동 처리"라고 표현한다.

## 7. 상태 설계

프로덕션 제품은 데이터가 있을 때만 제품처럼 보이면 안 된다. 모든 화면은 상태별로 안정적인 골격과 행동을 가져야 한다.

### 7.1 공통 상태

| 상태 | 의미 | UI 요구 |
| --- | --- | --- |
| Ready | 정상 데이터 표시 | 핵심 정보와 주요 액션 표시 |
| Empty | 결과 없음 | 짧은 빈 상태와 다음 행동 표시 |
| Loading | 불러오는 중 | 레이아웃을 유지한 상태에서 자리 표시 |
| Partial | 일부 정보 누락 | 누락 섹션만 빈 상태 처리 |
| Stale | 최신성 의심 | 업데이트 필요 라벨 표시 |
| Needs Review | 자동 판정 불충분 | 관리자 검토 필요 라벨 표시 |
| Unauthorized | 권한 없음 | 로그인 또는 계정 전환 안내 |
| Not Found | 대상 없음 | 탐색 경로 유지, 단독 에러 화면 금지 |
| Failed | 일시 실패 | 다시 시도 액션 표시 |

### 7.2 빈 상태 원칙

- 빈 상태는 최대 두 문장으로 끝낸다.
- 내부 구현 원인은 표시하지 않는다.
- 화면 골격은 유지한다.
- 최소 하나의 다음 행동을 제공한다.
- 관리자 화면의 빈 상태는 운영 메뉴와 문맥을 유지한다.

### 7.3 사용자에게 노출하지 않는 단어

공개/관리자 UI에서 다음 표현은 기본적으로 노출하지 않는다.

- API
- 서버
- 환경변수
- 토큰
- Bearer
- 응답
- 데이터
- endpoint
- URL
- PRD
- Issue
- Queue
- Agent

대체 표현:

| 내부 표현 | 사용자 표현 |
| --- | --- |
| API | 요청, 처리, 수집 |
| Server | 서비스, 처리 상태 |
| URL | 링크 |
| Queue | 목록, 검토 목록 |
| Agent | 자동 처리 |
| Cluster | 쟁점, 주장 묶음 |
| Data | 정보, 내용, 항목 |
| Token | 로그인 상태 |

문서와 API 명세에서는 내부 표현을 사용할 수 있지만, 앱 화면에 그대로 노출하면 안 된다.

## 8. 시각 및 인터랙션 시스템

### 8.1 레이아웃 원칙

- 랜딩 페이지식 히어로를 두지 않는다.
- 반복 항목 외에는 카드형 컨테이너를 기본 단위로 삼지 않는다.
- 정보는 선, 구분, 밀도, 계층으로 정리한다.
- 첫 화면은 바로 작업 화면이어야 한다.
- 제품 설명보다 현재 상태와 다음 행동이 먼저 보여야 한다.

### 8.2 표면 처리

- 배경은 흰색 또는 아주 낮은 대비의 중립 배경을 기본으로 한다.
- 중요 상태는 색상만으로 전달하지 않고 라벨과 숫자를 같이 쓴다.
- 둥근 카드 남발을 금지한다.
- 반복 목록은 `divide-y`, 테이블형 행, 세로 구분선 위주로 정리한다.

### 8.3 타이포그래피

- 제목은 짧고 기능적이어야 한다.
- 긴 설명 문구는 줄이고, 정보 라벨을 우선한다.
- 화면 안에서 "도움말 문장"이 정보보다 커지면 실패로 본다.

### 8.4 액션 원칙

- 모든 버튼은 실제 행동을 해야 한다.
- 행동이 불가능한 경우 비활성화 또는 명확한 상태 안내를 표시한다.
- 실패 메시지는 사용자 관점으로 쓴다.
- 내부 원인 대신 사용자가 할 수 있는 다음 행동을 말한다.

## 9. 데이터 및 연동 설계 기준

자세한 API 계약은 `docs/API_SPEC.md`를 따른다. 이 문서는 API 필드 자체보다 제품 화면에서 어떻게 해석할지를 정의한다.

### 9.1 조회 화면

- 홈, 이슈 상세, 관리자 콘솔은 조회 결과가 비어도 화면 골격을 유지한다.
- 각 섹션은 독립적으로 비어 있을 수 있다.
- 한 섹션이 비었다고 전체 화면을 실패 상태로 바꾸지 않는다.

### 9.2 생성/수정/삭제 행동

- 기사 검증 접수
- 주장 제출
- 저장/저장 해제
- 리포트 저장
- 신고 처리
- 출처 상태 변경
- 재검증 실행
- 출고 승인

각 행동은 성공, 진행 중, 실패 상태를 가져야 한다.

### 9.3 에러 매핑

| 내부 원인 | 사용자 메시지 |
| --- | --- |
| 연결 설정 없음 | 현재 요청을 처리할 수 없습니다. 잠시 후 다시 시도해 주세요. |
| 401/403 | 로그인 상태를 확인해 주세요. |
| 404 | 해당 항목을 찾을 수 없습니다. |
| 409 | 이미 처리된 항목입니다. |
| 422 | 입력한 내용을 다시 확인해 주세요. |
| 500 | 일시적으로 처리하지 못했습니다. 잠시 후 다시 시도해 주세요. |

## 10. 운영 및 안전 설계

### 10.1 판정 원칙

- 기사 전체를 가짜뉴스로 단정하지 않는다.
- 주장 단위로 판정한다.
- 후속 정보가 나오면 초기 기준과 최신 기준을 분리한다.
- 법적 쟁점은 자동으로 결론 내리지 않는다.
- 확인 불가능한 고의성 주장은 근거 부족으로 분리한다.

### 10.2 낙인 표현 처리

낙인 표현은 원문 그대로 확산하지 않는다.

처리 기준:

- 모욕/비방 표현 감지
- 검증 가능한 주장과 감정 표현 분리
- 정제 표현 생성
- 원문은 관리자 검토 화면에서만 제한적으로 확인
- 공개 화면에는 정제된 주장과 검증 상태만 표시

### 10.3 출고 전 확인

관리자는 공개 전 다음을 확인한다.

- 최신 공식 자료 반영 여부
- 수치 기준과 시점 분리 여부
- 낙인 표현 정제 여부
- 검증 어려운 주장에 단정 표현이 없는지
- 출처 신뢰도와 원문 링크 확인 여부
- 사용자 제출 주장 중 중복/악성 제보 제거 여부

## 11. 프로덕션 준비 우선순위

### 11.1 1차 필수

- 홈 빈 상태의 제품 골격 유지
- 관리자 상세 빈 상태 재설계
- 사용자 노출 문구에서 내부 기술어 제거
- 로그인/회원가입 실제 흐름
- 기사 검증 접수와 주장 제출의 성공/실패 상태
- 관리자 검토 목록과 상세 연동
- API 에러 매핑

### 11.2 2차 필수

- 이슈 상세 탭 구조
- 기사별 비교 고도화
- 원문 자료 신뢰도 표시
- 사용자 계정 대시보드
- 출처 관리 상태 변경
- 신고 처리 이력

### 11.3 3차 확장

- 네이티브 모바일 앱 알림
- 추가 리포트 내보내기 형식
- 관심 관점 개인화
- 타임라인 업데이트 알림
- 수치 변경 추적 히스토리

## 12. 수용 기준

프로덕션으로 보기 위한 최소 기준은 다음과 같다.

- 공개 웹, 이슈 상세, 관리자 메인, 관리자 상세, 로그인, 내 계정이 모두 독립적으로 탐색 가능하다.
- 데이터가 없는 상태에서도 각 화면이 제품 골격과 다음 행동을 유지한다.
- 사용자 화면에 내부 기술어가 노출되지 않는다.
- 모든 버튼은 실제 행동, 비활성 상태, 또는 명확한 실패 메시지를 가진다.
- 더미 데이터 없이 실제 연동 준비 상태로 동작한다.
- 화면은 모바일 390px과 데스크톱 1440px에서 가로 스크롤이 없어야 한다.
- 낙인 표현은 공개 화면에서 원문 그대로 확산되지 않는다.
- 관리자 상세는 데이터를 찾지 못해도 단독 에러 페이지처럼 보이면 안 된다.
- API 명세와 화면 요구 필드가 서로 모순되지 않는다.

## 13. 현재 구현 대비 주요 차이

현재 구현은 공개 웹, 이슈 상세 탭, 직접 검증, 사용자 참여, 관리자, 리포트, PWA 기본 셸, API 연결 구조를 갖춘다.

프로덕션 확장으로 남겨 둔 범위는 다음이다.

- 네이티브 모바일 앱, 공유 확장, 모바일 전용 푸시 경험은 이번 범위에서 제외한다.
- 외부 수집, LLM, 푸시 제공자, 배포 인프라는 실제 운영 키와 환경 설정이 필요하다.
- 대규모 트래픽 기준의 검색 색인, 관측성, 장애 알림은 운영 단계에서 고도화한다.

이 차이는 결함 목록이 아니라 구현 우선순위다. 이후 화면 변경은 이 문서의 수용 기준을 기준으로 판단한다.
